# Phase 10 설계: JsonStore 인메모리 캐싱 (성능 최적화)

## 배경

스트레스 테스트(주문 1000건 규모)에서 매 호출마다 JSON 파일 전체를 다시 읽고 쓰는 구조 때문에
처리 시간이 선형이 아닌 그 이상으로 증가하는 것을 확인함(1000건 승인에 약 17.8초).

## 변경 범위

`app/repositories/json_store.py`만 수정한다. 외부 인터페이스(`find_all`, `find_by_id`,
`find_by_field`, `insert`, `update`, `delete`)의 시그니처와 동작 계약은 변경하지 않는다.

## 설계

- `JsonStore.__init__`에서 파일을 읽어 `self._cache: list[dict]`에 보관한다(최초 1회만 파일 읽기).
- `find_all()`/`find_by_id()`/`find_by_field()`는 캐시를 직접 참조하여 반환한다(파일 재읽기 없음).
  단, 호출자가 반환된 dict를 직접 변경해 캐시를 오염시키지 않도록 **얕은 복사본**(각 record의
  사본)을 반환한다.
- `insert()`/`update()`/`delete()`는 캐시를 갱신한 뒤 즉시 파일에 반영한다(쓰기 시점 데이터
  유실 방지 원칙 유지, 매 쓰기마다 디스크 반영은 그대로 유지).
- 여러 `JsonStore` 인스턴스가 같은 파일을 동시에 여는 시나리오(예: 다른 프로세스)는 캐시
  일관성이 깨질 수 있다. 이는 CLAUDE.md에 이미 명시된 "범위 밖 한계"(멀티프로세스 동시성
  미대응)와 동일선상이므로 새로운 제약이 아니다.

## 테스트 전략

- 기존 `tests/test_json_store.py`, `test_sample_repository.py`, `test_order_repository.py`,
  `test_persistence.py`가 모두 그대로 통과해야 한다(동작 계약 불변 검증).
- 캐시가 실제로 파일 재읽기를 생략하는지 확인하는 테스트를 추가한다(파일을 캐시 생성 후
  외부에서 직접 변조해도 `find_all()` 결과가 바뀌지 않아야 함 - 캐시 사용 검증용).
- 반환된 dict를 호출자가 변경해도 내부 캐시가 오염되지 않는지 확인하는 테스트를 추가한다.
- 스트레스 테스트를 재실행하여 처리 시간이 유의미하게 감소하는지 수동으로 확인한다(자동
  성능 회귀 테스트로 시간 임계값을 넣지는 않음 - 환경마다 편차가 크므로).
