# 개발 계획서 (PLAN.md)

## 개발 철학

각 Phase는 독립적으로 실행/테스트 가능한 완성된 소프트웨어 단위로 진행한다.
Phase 완료 시마다 사용자가 직접 실행 및 pytest로 검증하고, 승인 후 다음 Phase로 진행한다.
(참고: 본 PLAN.md는 개발 진행 상황 추적용 문서이며, 완료 후에도 AE 프로세스 이력 증빙을 위해 리포지토리에 유지한다.)

### Phase 진행 절차 (Phase 7부터 적용)

1. AI가 `docs/design/phaseN.md` 설계 문서를 작성하여 제시한다.
2. 사람이 설계 문서를 검토하고, 필요 시 수정을 요청한다 (기대치 확보).
3. 승인된 설계 문서를 기반으로만 AI가 구현(TDD RED→GREEN)을 진행한다.
4. Verify Harness(`scripts/verify.py`) 통과 후 사람이 콘솔 E2E 검증 및 최종 리뷰.
5. 설계와 실제 구현이 달라진 부분이 있다면 설계 문서에도 반영(devation 기록)한 뒤 커밋한다.

## Phase 목록

- [x] Phase 1: 프로젝트 스캐폴드 + 데이터 영속성 계층
  - Sample/Order 모델, JsonStore 기반 Repository, 초기 데이터 없이도 정상 기동
  - 테스트 포인트: pytest로 CRUD 동작 확인, 프로세스 재시작 후 데이터 유지 확인
- [x] Phase 2: 메인 메뉴 + 시료 관리
  - 메인 메뉴 뼈대(선택 → 하위 메뉴 이동 → 종료), 시료 등록/조회/검색
  - 테스트 포인트: 시료 등록 후 조회 시 반영되는지, 검색 결과 정확성
- [x] Phase 3: 시료 주문(예약)
  - 시료 ID/고객명/수량 입력 → RESERVED 주문 생성
  - 테스트 포인트: 존재하지 않는 시료 ID 주문 시 처리, 주문번호 자동 채번
- [x] Phase 4: 주문 승인/거절
  - RESERVED 목록 표시, 승인 시 재고 비교 후 CONFIRMED/PRODUCING 분기 및 생산 큐 등록, 거절 시 REJECTED
  - 테스트 포인트: 재고 충분/부족 경계값(재고==주문수량) 처리, 거절 후 상태 불변성
- [x] Phase 5: 생산 라인
  - 생산 큐 FIFO 처리, 실 생산량/생산 시간 계산, 생산 완료 시 PRODUCING→CONFIRMED 및 재고 반영, 생산 현황/대기열 조회
  - 테스트 포인트: 수율에 따른 실 생산량 반올림(ceil) 정확성, FIFO 순서 보장
- [x] Phase 6: 모니터링
  - 상태별 주문 수(REJECTED 제외), 시료별 재고 상태(여유/부족/고갈) 조회
  - 테스트 포인트: 경계값(재고 0 → 고갈) 처리
- [x] Phase 7: 출고 처리
  - CONFIRMED 주문 목록 표시 및 출고 실행 → RELEASE 전환
  - 테스트 포인트: CONFIRMED가 아닌 주문은 출고 목록에 노출되지 않는지
- [ ] Phase 8: 통합 마감
  - 전체 메뉴 연결 점검, 회귀 테스트 전체 재실행, README 정리, 더미 데이터 시나리오로 End-to-End 수동 검증

## Phase별 테스트 체크리스트 공통 항목

각 Phase 완료 시 다음을 확인한다.
1. 신규 기능이 요구사항(PRD.md)을 충족하는가
2. 기존 pytest 스위트가 모두 PASS하는가 (Regression)
3. 콘솔에서 직접 실행하여 실제 동작을 확인했는가
